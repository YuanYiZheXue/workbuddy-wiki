# LLM Wiki

一种利用 LLM 构建个人知识库的模式。

这是一个创意文件，设计用于复制粘贴到你自己的 LLM Agent（如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目标是传达高层理念，但具体细节由你的 Agent 与你协作构建。

## 核心理念

大多数人与 LLM 和文档交互的体验类似于 RAG：你上传一批文件，LLM 在查询时检索相关片段，然后生成答案。这能工作，但 LLM 对每个问题都是从零开始重新发现知识，没有积累。如果你问一个需要综合五篇文档的微妙问题，LLM 每次都要重新找到并拼凑相关片段，什么都没有建立起来。NotebookLM、ChatGPT 文件上传，以及大多数 RAG 系统都是这样工作的。

这里的理念不同。不是在查询时仅从原始文档中检索，而是让 LLM **增量构建并维护一个持久化的 Wiki**——一个结构化的、相互链接的 Markdown 文件集合，位于你和原始资料之间。当你添加一个新的资料来源时，LLM 不只是为其建立索引以供日后检索，而是阅读它、提取关键信息，并将其整合到现有 Wiki 中——更新实体页面、修订主题摘要、注明新数据与旧主张的矛盾之处、强化或挑战不断演进的综合结论。知识是被编译一次然后*保持最新*，而不是每次查询都重新推导。

这是关键区别：**Wiki 是一个持久的、具有复合效应的产物。** 交叉引用已经存在。矛盾已经被标记。综合结论已经反映了你阅读过的所有内容。每添加一份资料、每提出一个问题，Wiki 都会变得更加丰富。

你从不（或很少）自己写 Wiki——LLM 编写并维护其中的所有内容。你负责资料来源、探索和方向、提出正确的问题。LLM 做所有的苦力活——摘要、交叉引用、归档和簿记，这些工作使得知识库随着时间推移真正有用。在实践中，我把 LLM Agent 放在一边，Obsidian 放在另一边。LLM 根据我们的对话进行编辑，我实时浏览结果——跟随链接、查看图谱视图、阅读更新的页面。Obsidian 是 IDE；LLM 是程序员；Wiki 是代码库。

这可以应用于许多不同的场景。几个例子：

- **个人**：追踪你自己的目标、健康、心理、自我提升——归档日记条目、文章、播客笔记，并随着时间推移建立关于你自己的结构化画像。
- **研究**：在数周或数月内深入研究一个主题——阅读论文、文章、报告，并逐步建立一个带有演进论点的全面 Wiki。
- **阅读一本书**：随着阅读进度归档每一章，为人物、主题、情节线索及其联系建立页面。到读完时，你就拥有了一个丰富的配套 Wiki。想想像[Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page)这样的粉丝 Wiki——由志愿者社区经过多年努力建立的数千个相互链接的页面，涵盖人物、地点、事件、语言。你可以在阅读时个人建立类似的东西，由 LLM 完成所有交叉引用和维护工作。
- **商业/团队**：由 LLM 维护的内部 Wiki，输入来自 Slack 线程、会议记录、项目文档、客户电话。可能有人参与审核更新。Wiki 保持最新，因为 LLM 做了团队中没人想做的维护工作。
- **竞品分析、尽职调查、行程规划、课程笔记、爱好深入研究**——任何你需要随着时间积累知识并希望将其组织起来而不是散落各处的场景。

## 架构

有三层：

**原始资料**——你精心策划的源文档集合。文章、论文、图片、数据文件。这些是不可变的——LLM 从中读取但从不修改它们。这是你的真理来源。

**Wiki**——一个由 LLM 生成的 Markdown 文件目录。摘要、实体页面、概念页面、对比、概览、综合结论。LLM 完全拥有这一层。它创建页面、在新资料到达时更新它们、维护交叉引用，并保持一切一致。你阅读它；LLM 写它。

**Schema**——一个文档（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM Wiki 是如何结构的、约定是什么、以及在摄取资料、回答问题或维护 Wiki 时要遵循什么工作流。这是关键的配置文件——它使得 LLM 成为一个有纪律的 Wiki 维护者，而不是一个通用的聊天机器人。你和 LLM 随着时间的推移共同演进这个文件，因为你们会发现什么对你的领域有效。

## 操作

**摄取（Ingest）。** 你将一个新的资料来源放入原始集合，并告诉 LLM 处理它。一个示例流程：LLM 阅读资料来源，与你讨论关键要点，在 Wiki 中写一个摘要页面，更新索引，更新 Wiki 中各相关的实体和概念页面，并追加一个条目到日志。一个单一的资料来源可能触及 10-15 个 Wiki 页面。就个人而言，我更喜欢一次摄取一个资料并保持参与——我阅读摘要，检查更新，并指导 LLM 强调什么。但你也可以一次性批量摄取许多资料，减少监督。由你来开发适合你风格的工作流，并将其记录在 Schema 中以供未来会话使用。

**查询（Query）。** 你针对 Wiki 提问。LLM 搜索相关页面，阅读它们，并综合一个带有引用的答案。根据问题的不同，答案可以采取不同的形式——一个 Markdown 页面、一个对比表格、一套幻灯片（Marp）、一张图表（matplotlib）、一个画布。重要的见解：**好的答案可以作为新页面归档回 Wiki。** 你要求的对比、一个分析、一个你发现的联系——这些都是有价值的，不应该消失在聊天历史中。这样，你的探索就像摄取的资料一样在知识库中复合增长。

**整理（Lint）。** 定期要求 LLM 对 Wiki 进行健康检查。查找：页面之间的矛盾、已被更新的资料来源取代的过时主张、没有入站链接的孤儿页面、被提到但缺少自己页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 擅长建议新的调查问题和寻找新的资料来源。这使得 Wiki 在增长时保持健康发展。

## 索引和日志

两个特殊文件帮助 LLM（和你）在 Wiki 增长时导航。它们服务于不同的目的：

**index.md** 是面向内容的。它是 Wiki 中一切内容的目录——每个页面列出一个链接、一行摘要，以及可选的元数据如日期或来源数量。按类别组织（实体、概念、来源等）。LLM 在每次摄取时更新它。在回答查询时，LLM 首先阅读索引以找到相关页面，然后深入其中。这在中等规模（约 100 个来源，约数百个页面）下效果出奇地好，并避免了对基于嵌入的 RAG 基础设施的需求。

**log.md** 是按时间顺序排列的。它是一个仅追加的记录，记录发生了什么以及何时发生——摄取、查询、整理操作。一个有用的提示：如果每个条目都以一致的前缀开头（例如 `## [2026-04-02] ingest | Article Title`），那么日志可以用简单的 unix 工具解析——`grep "^## \[" log.md | tail -5` 给你最后 5 个条目。日志给你 Wiki 演进的时间线，并帮助 LLM 了解最近做了什么。

## 可选：CLI 工具

在某些时候，你可能想要构建小型工具来帮助 LLM 更高效地操作 Wiki。最明显的一个是 Wiki 页面上的搜索引擎——在小规模下索引文件就足够了，但随着 Wiki 的增长，你需要适当的搜索。[qmd](https://github.com/tobi/qmd) 是一个很好的选择：它是一个用于 Markdown 文件的本地搜索引擎，具有混合 BM25/向量搜索和 LLM 重排序，全部在设备上。它既有 CLI（所以 LLM 可以调用它），也有 MCP 服务器（所以 LLM 可以把它作为原生工具使用）。你也可以自己构建更简单的东西——LLM 可以帮助你根据需要对一个朴素的搜索脚本进行 vibe-coding。

## 提示和技巧

- **Obsidian Web Clipper** 是一个将网络文章转换为 Markdown 的浏览器扩展。对于快速将资料来源放入你的原始集合非常有用。
- **本地下载图片。** 在 Obsidian 设置 → 文件和链接中，将"附件文件夹路径"设置为一个固定目录（例如 `raw/assets/`）。然后在设置 → 热键中，搜索"Download"以找到"下载当前文件的附件"并将其绑定到一个热键（例如 Ctrl+Shift+D）。在剪辑一篇文章后，按下热键，所有图片都会被下载到本地磁盘。这是可选的但很有用——它让 LLM 可以直接查看和引用图片，而不是依赖可能失效的 URL。注意，LLM 无法在一次传递中原生读取带有内联图片的 Markdown——变通方法是让 LLM 先读取文本，然后单独查看部分或所有引用的图片以获得额外的上下文。这有点笨拙但效果足够好。
- **Obsidian 的图谱视图**是查看 Wiki 形状的最佳方式——什么与什么相连，哪些页面是枢纽，哪些是孤儿。
- **Marp** 是一种基于 Markdown 的幻灯片格式。Obsidian 有一个插件支持它。用于直接从 Wiki 内容生成演示文稿很有用。
- **Dataview** 是一个 Obsidian 插件，可以对页面 frontmatter 运行查询。如果你的 LLM 向 Wiki 页面添加 YAML frontmatter（标签、日期、来源数量），Dataview 可以生成动态表格和列表。
- Wiki 只是一个 Markdown 文件的 git 仓库。你免费获得版本历史、分支和协作。

## 为什么这有效

维护知识库的乏味部分不是阅读或思考——而是簿记。更新交叉引用、保持摘要最新、注意新数据何时与旧主张矛盾、在数十个页面间保持一致性。人类放弃 Wiki 是因为维护负担增长得比价值快。LLM 不会感到无聊，不会忘记更新交叉引用，并且可以在一次传递中触及 15 个文件。Wiki 保持维护是因为维护成本接近零。

人类的工作是策划资料来源、指导分析、提出好问题，并思考这一切意味着什么。LLM 的工作是一切其他事情。

这个理念在精神上类似于 Vannevar Bush 的 Memex（1945）——一个个人的、精心策划的知识存储，文档之间有联想线索。Bush 的愿景更接近于这个而不是网络变成的样子：私有的、积极策划的，文档之间的连接与文档本身一样有价值。他无法解决的部分是谁来做维护。LLM 处理了这一点。

## 注意

本文档是有意抽象的。它描述的是理念，而不是具体的实现。确切的目录结构、Schema 约定、页面格式、工具——所有这些都取决于你的领域、你的偏好和你选择的 LLM。上面提到的一切都是可选和模块化的——选择有用的，忽略没用的。例如：你的资料来源可能只是纯文本的，所以你根本不需要图片处理。你的 Wiki 可能足够小，索引文件就是你所需要的一切，不需要搜索引擎。你可能不关心幻灯片，只想要 Markdown 页面。你可能想要一套完全不同的输出格式。使用它的正确方法是与你的 LLM Agent 分享它，并共同工作以实例化一个适合你需求的版本。本文档的唯一工作是传达模式。你的 LLM 可以弄清楚其余的事情。
