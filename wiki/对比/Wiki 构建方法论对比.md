# Wiki 构建方法论对比

> 对比 llm-wiki（原始理念）、元一思想体系（哲学升级版）、workbuddy-wiki-schema.md（实际实现）三种 Wiki 构建方法论。

## 对比维度

| 维度 | 说明 |
|------|------|
| 哲学基础 | 构建 Wiki 的底层逻辑 |
| Schema 设计 | 如何定义和规范 Wiki 结构 |
| 操作流程 | ingest / query / lint 的具体实现 |
| 健康度诊断 | 如何评估 Wiki 是否健康 |
| 跨会话持久化 | 如何保持跨会话的连续性 |

---

## 详细对比

| 方法论 | 哲学基础 | Schema 设计 | 操作流程 | 健康度诊断 | 跨会话持久化 |
|----------|----------|----------|----------|--------------|----------------|
| **llm-wiki（原始理念）** | 无明确哲学；实用主义（让 LLM 增量构建 Wiki） | 三层架构：原始资料 / Wiki / Schema | ingest（读取资料 → 更新 Wiki）、query（搜索 → 综合）、lint（健康检查） | 无明确诊断模型；依赖 LLM 自我检查 | Schema 文件（CLAUDE.md / AGENTS.md） |
| **元一思想体系** | 元一思想四原则（存续/流动/意义/结构） + 四极飞轮模型 | Schema 以四原则为顶层设计；所有决策用四原则审视 | 与 llm-wiki 相同，但每一步都用四原则审视 | 四极飞轮模型（四大核心制衡）；定期诊断并调优 | Schema 文件 + MEMORY.md（跨会话记忆） |
| **workbuddy-wiki-schema.md（实际实现）** | 元一思想四原则 + 接口决策规则 + 中文版独有章节 | 在元一思想基础上，添加接口决策规则、中文版独有章节 | 与元一思想体系相同，但更详细（如"先3后2再1"工作流） | 四极飞轮模型 + 三轮健康度诊断（已执行） | Schema 文件 + MEMORY.md + .workbuddy/ 目录 |

---

## 关键洞察

### 1. 哲学基础是 Wiki 的"灵魂"

> "本文档是地图，不是百科全书——按需扩展，不追求一次完美。"—— workbuddy-wiki-schema.md

| 方法论 | 哲学基础 | 优势 | 局限 |
|----------|----------|------|------|
| llm-wiki | 无（实用主义） | 简单、易上手 | 缺少价值判断标准，迭代容易迷失方向 |
| 元一思想体系 | 四原则 + 四极飞轮 | 有灵魂、有标准、可诊断 | 需要用户理解并接受这套哲学 |
| workbuddy-wiki-schema.md | 元一思想 + 接口决策规则 | 哲学 + 可操作规则，兼顾灵魂与实操 | 复杂度较高，需要一定学习成本 |

### 2. Schema 设计的演进：从"是什么"到"为什么"

| 版本 | Schema 设计特点 | 代表文件 |
|------|----------------|----------|
| v0.1.0 | 描述"是什么"（三层架构、操作流程） | llm-wiki.md |
| v0.2.0 | 添加"为什么"（元一思想四原则作为顶层设计） | 元一思想体系 + workbuddy-wiki-schema.md（初版） |
| v0.3.0 | 添加"怎么判断"（接口决策规则、中文版独有章节） | workbuddy-wiki-schema.md（当前版） |

**演进逻辑**：
- v0.1.0：告诉 LLM "做什么"（构建 Wiki）
- v0.2.0：告诉 LLM "为什么这么做"（元一思想）
- v0.3.0：告诉 LLM "怎么判断"（接口决策规则）

### 3. 健康度诊断：从"感觉"到"模型"

| 方法论 | 健康度诊断 | 特点 |
|----------|--------------|------|
| llm-wiki | 无明确诊断模型 | 依赖 LLM 自我检查，不稳定 |
| 元一思想体系 | 四极飞轮模型（四大核心制衡） | 有模型、可操作，但需要人工执行 |
| workbuddy-wiki-schema.md | 四极飞轮模型 + 三轮健康度诊断（已执行） | 模型 + 实际案例，可复现 |

**诊断实例**（来自 workbuddy-wiki-schema.md）：
- 第一轮诊断：发现 Schema 有过度设计倾向、来源页→概念页的双向链接不完整
- 第二轮诊断：发现 `LangChain.md` 有悬空链接、接口决策规则未经验证
- 第三轮诊断：发现 `README.md` 版本信息未更新、缺少对比页

### 4. 跨会话持久化的实现差异

| 方法论 | 跨会话持久化机制 | 优势 | 局限 |
|----------|----------------|------|------|
| llm-wiki | Schema 文件（CLAUDE.md / AGENTS.md） | 简单、原生支持 | 只持久化 Schema，不持久化"为什么这么做" |
| 元一思想体系 | Schema 文件 + MEMORY.md | 持久化 Schema + 跨会话记忆 | MEMORY.md 需要人工维护 |
| workbuddy-wiki-schema.md | Schema 文件 + MEMORY.md + .workbuddy/ 目录 | 持久化 Schema + 记忆 + 工作日志 | 目录结构较复杂 |

**workbuddy 特有优势**：
- `.workbuddy/memory/YYYY-MM-DD.md`：每日工作记录，自动追加
- `.workbuddy/memory/MEMORY.md`：长期记忆，手动更新
- `.workbuddy/schema-update.md`：Schema 更新流程，手动触发

## 与元一思想的映射

| Wiki 构建实践 | 元一思想对应 | 解释 |
|----------------|----------------|------|
| 增量构建 Wiki（不一次性完美） | **存续为体，形式为用** | Wiki 的价值在于被使用，不是为了完美而存在 |
| ingest → query → lint 循环 | **流动趋效，均衡为度** | 信息要高效流动，但以不过载为限度 |
| 健康度诊断（四极飞轮） | **意义生于博弈，固于认同** | 诊断发现的问题推动 Wiki 演进（博弈），形成稳定的构建规范（认同） |
| Schema 版本化管理 | **结构求稳，接口预变** | Schema 版本化（结构求稳），接口决策规则（接口预变） |
