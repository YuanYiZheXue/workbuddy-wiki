# 模型选择策略对比

> 汲取自 `wiki/来源/` 中多篇素材的综合对比。

## 引言

模型选择策略直接影响编码 Agent 的性能、成本和用户体验。本文对比 Anthropic、OpenDev、LangChain、Codex 四大主流框架在模型选择策略上的设计选择。

---

## 对比维度

| 维度 | Anthropic | OpenDev | LangChain | Codex |
|------|-----------|---------|-----------|-------|
| **默认模型** | Claude 3 Opus/Sonnet | GPT-4/GPT-3.5 | 无（用户指定） | Codex |
| **模型切换策略** | 手动选择 | 自动选择（根据任务） | 手动/自动 | 不支持 |
| **成本优化** | 无原生支持 | 无原生支持 | 有（成本追踪） | 无原生支持 |
| **性能优化** | 无原生支持 | 无原生支持 | 有（性能监控） | 无原生支持 |
| **多模型编排** | 不支持 | 不支持 | 支持 | 不支持 |
| **本地模型支持** | 不支持 | 不支持 | 支持 | 不支持 |

---

## 分析

### Anthropic：固定+手动

**策略**：使用 Claude 系列模型，手动选择具体模型。

**实现**：
```
用户选择模型（Opus/Sonnet/Haiku）
    ↓
Claude API 调用
    ↓
返回结果
```

**优势**：
- 简单、可控
- 模型性能稳定（Claude 3 系列质量高）

**劣势**：
- 不灵活（无法自动切换）
- 无成本优化（所有请求都用高价模型）

**适用场景**：需要稳定性能的生产环境。

---

### OpenDev：自动选择

**策略**：根据任务类型自动选择模型。

**实现**：
```
用户请求
    ↓
OpenDev 平台（自动选择模型）
    ↓
GPT-4/GPT-3.5 API 调用
    ↓
返回结果
```

**优势**：
- 简单、自动
- 成本优化（简单任务用低价模型）

**劣势**：
- 不透明（用户不知道用了哪个模型）
- 依赖平台（无法自定义策略）

**适用场景**：快速原型、依赖 OpenDev 生态的应用。

---

### LangChain：灵活编排

**策略**：支持多种模型，可手动或自动切换。

**实现**：
```
用户选择/自动选择模型
    ↓
LCEL 编排
    ↓
多种模型 API 调用（GPT-4、Claude、Llama 等）
    ↓
返回结果
```

**优势**：
- 灵活、可优化
- 成本优化（成本追踪模块）
- 性能优化（性能监控模块）
- 多模型编排（Chain 中可切换模型）

**劣势**：
- 复杂（需要配置 LCEL）
- 依赖外部 API（需要管理多个 API Key）

**适用场景**：需要灵活编排、成本优化的研究环境。

---

### Codex：固定模型

**策略**：使用 Codex 模型，不支持切换。

**实现**：
```
用户请求
    ↓
Codex API 调用
    ↓
返回结果（代码）
```

**优势**：
- 简单（无需选择模型）
- 代码生成质量高（Codex 专门训练）

**劣势**：
- 不灵活（无法切换模型）
- 无成本优化（所有请求都用 Codex）
- 无性能监控（无法追踪性能）

**适用场景**：代码生成、代码执行任务。

## 核心差异

### 1. 模型切换策略

| 框架 | 切换策略 | 优势 | 劣势 |
|------|--------------|------|------|
| Anthropic | 手动选择 | 可控 | 不灵活 |
| OpenDev | 自动选择 | 简单 | 不透明 |
| LangChain | 手动/自动 | 灵活 | 复杂 |
| Codex | 不支持 | 简单 | 不灵活 |

### 2. 成本优化能力

| 框架 | 成本优化 | 实现方式 |
|------|----------|----------|
| Anthropic | 无 | - |
| OpenDev | 无 | - |
| LangChain | 有 | 成本追踪模块（追踪每个请求的成本） |
| Codex | 无 | - |

### 3. 多模型编排能力

| 框架 | 多模型编排 | 实现方式 |
|------|--------------|----------|
| Anthropic | 不支持 | - |
| OpenDev | 不支持 | - |
| LangChain | 支持 | LCEL 编排（Chain 中可切换模型） |
| Codex | 不支持 | - |

---

## 选择建议

### 选择 Anthropic 的场景

- 需要稳定性能的生产环境
- 不需要成本优化
- 不需要灵活编排

### 选择 OpenDev 的场景

- 快速原型
- 依赖 OpenDev 生态
- 不需要透明模型选择

### 选择 LangChain 的场景

- 需要灵活编排
- 需要成本优化
- 需要性能监控
- 研究环境

### 选择 Codex 的场景

- 代码生成、代码执行任务
- 不需要灵活编排
- 不需要成本优化

## 元一思想视角

用**元一思想**四原则分析：

| 原则 | 分析 |
|------|------|
| **存续为体，形式为用** | LangChain 的灵活编排最符合——形式（多模型）服务于存续（任务完成），但不过度设计。 |
| **流动趋效，均衡为度** | OpenDev 的自动选择最快速——信息流动快，但容易不透明（用户不知道用了哪个模型）。需要均衡。 |
| **意义生于博弈，固于认同** | Anthropic 的稳定性能最可靠——意义在与用户的博弈中产生，但需要认同手动选择约束。 |
| **结构求稳，接口预变** | LangChain 的 LCEL 最稳定——结构（LCEL 编排）不变，接口（模型 API）可扩展。 |

**推荐选择**：
- **生产环境**：Anthropic（稳定性能）
- **研究环境**：LangChain（灵活编排）
- **快速原型**：OpenDev（自动选择）
- **代码任务**：Codex（代码生成）
