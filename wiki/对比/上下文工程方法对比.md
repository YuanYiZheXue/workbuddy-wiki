# 上下文工程方法对比

> 对比 Anthropic、OpenDev、LangChain、Codex 四个框架的上下文工程管理策略。

## 对比维度

| 维度 | 说明 |
|------|------|
| 上下文管理策略 | 如何组织、更新、压缩上下文窗口 |
| 上下文压缩方法 | 上下文窗口快满时如何处理 |
| 长视野处理 | 跨多轮会话如何保持上下文连续性 |
| 工具选择对上下文的影响 | 工具数量/描述如何影响上下文占用 |
| 评估方法 | 如何评估上下文工程的效果 |

## 详细对比

| 框架 | 上下文管理策略 | 上下文压缩方法 | 长视野处理 | 工具选择对上下文的影响 | 评估方法 |
|--------|----------------|----------------|--------------|--------------------------|----------|
| **Anthropic** | 双 Agent 架构：Initializer 建立 progress file + feature list；Coding Agent 每次只做一个功能 | 未明确说明压缩方法；依赖 git commit 和 progress file 作为外部记忆 | progress file（增量进展）+ feature list（功能清单）+ git log（历史记录） | 未明确讨论；但强调"一次只做一个功能"以减少上下文占用 | 端到端测试（Puppeteer MCP 截图验证） |
| **OpenDev** | 上下文工程作为一等公民；四层架构中的 Tool & Context Layer | Adaptive Context Compaction（自适应上下文压缩）：多阶段压缩管道，逐步压缩旧 observation | Session 持久化（Conversation History + Provider Cache + Operation Log） | Lazy Tool Discovery（惰性工具发现）：只加载相关工具描述，减少上下文占用 | Terminal Bench（Agent 基准测试平台） |
| **LangChain** | Harness 负责上下文管理；Agent = Model + Harness | 未明确说明；依赖 Harness 设计 | Harness 可以包含持久化机制（如 Memory 模块） | 工具注册表由 Harness 管理；工具数量直接影响上下文占用 | LangSmith（评估、追踪、调试平台） |
| **Codex** | 小而精的 Harness；AGENTS.md 作为上下文核心 | 未明确说明；依赖模型原生上下文窗口 | AGENTS.md 作为跨会话的持久化上下文 | 工具选择极简；AGENTS.md 中只描述必要工具 | 未明确说明；强调"适度原则" |

## 关键洞察

### 1. 上下文工程 ≠ 更大的上下文窗口

> "更大的上下文窗口不等于更好的性能（噪声更多 = 上下文腐烂更快）"—— Louis Bouchard

四个框架都认同：上下文工程的核心是**选择性输入**，而非"把所有信息都塞进去"。

### 2. 持久化机制是长视野 Agent 的关键

| 框架 | 持久化机制 | 优势 | 局限 |
|--------|--------------|------|------|
| Anthropic | progress file + feature list + git commit | 人类可读、可审查、可回滚 | 需要额外 token 维护这些文件 |
| OpenDev | Session 持久化（Conversation History + Provider Cache + Operation Log） | 自动化程度高；利用 Provider Cache 降低成本 | 依赖特定 Provider 的缓存机制 |
| LangChain | Harness 可以包含 Memory 模块 | 灵活；可自定义持久化策略 | 需要开发者自行设计 |
| Codex | AGENTS.md（跨会话持久化） | 极简；人类可编辑 | 只适合小规模项目 |

### 3. 工具选择对上下文的影响被低估了

- **Anthropic**：未明确讨论，但"一次只做一个功能"间接减少了工具调用次数
- **OpenDev**：Lazy Tool Discovery（惰性工具发现）—— 只加载相关工具描述
- **LangChain**：工具注册表由 Harness 管理；工具数量直接影响上下文占用
- **Codex**：工具选择极简；AGENTS.md 中只描述必要工具

**共同趋势**：工具描述正在从"全量加载"转向"按需加载"。

### 4. 评估方法差异巨大

| 框架 | 评估方法 | 特点 |
|--------|----------|------|
| Anthropic | 端到端测试（Puppeteer MCP 截图验证） | 黑盒测试；验证功能是否真正工作 |
| OpenDev | Terminal Bench（Agent 基准测试平台） | 量化评估；可横向对比不同 Agent 框架 |
| LangChain | LangSmith（评估、追踪、调试平台） | 白盒测试；可追踪每次工具调用和模型推理 |
| Codex | 未明确说明；强调"适度原则" | 可能依赖人工审查 |

## 与元一思想的映射

| 上下文工程实践 | 元一思想对应 | 解释 |
|----------------|----------------|------|
| 选择性输入（不把所有信息都塞进去） | **流动趋效，均衡为度** | 信息要高效流动，但以不过载为限度 |
| 持久化机制（progress file、Session 持久化） | **存续为体，形式为用** | 知识库/ Agent 的价值在于被使用，持久化是"存续"的手段 |
| 工具按需加载（Lazy Tool Discovery） | **结构求稳，接口预变** | 工具是"接口"，按需加载是"预变"的体现 |
| 根据错误改进 Harness（Anthropic 的实践） | **意义生于博弈，固于认同** | 错误是"博弈"，改进是"认同"的固化 |
