# Agent 评估方法对比

> 不同框架如何评估 Agent 的性能、正确性和用户体验。

## 对比维度

| 维度 | Anthropic | OpenDev | LangChain | Codex |
|------|-----------|---------|-----------|-------|
| **评估方法** | Human eval + Evaluator Agent | 自动化测试 + 用户反馈 | LangSmith Tracing + 自定义指标 | 单元测试 + 集成测试 |
| **评估粒度** | 每个 sprint 结束 | 每个 task 结束 | 每个 chain 调用 | 每个 PR |
| **评估标准** | 硬编码阈值 + 人工审查 | 测试用例通过率 | 自定义指标（准确性、延迟、成本） | 代码覆盖率 + 功能正确性 |
| **可观测性** | Trace 分析（手动） | Dashboard（实时） | LangSmith Traces（自动） | GitHub Actions（CI/CD） |
| **反馈闭环** | 手动回顾 | 自动反馈 | 手动 + 自动 | 手动 |

---

## 详细对比

### 1. Anthropic 评估方法

**核心思想**：分离生成与评估（Generator + Evaluator）

**评估流程**：
1. Generator Agent 完成一个 sprint
2. Evaluator Agent 验证（使用 Playwright MCP 直接与实时页面交互）
3. 每条评估标准有硬性阈值，任一条不达标则 sprint 失败
4. Sprint Contract：generator 和 evaluator 在写代码前先协商"完成"的定义

**优势**：
- 评估者变得"怀疑"，比让生成器批判自己的作品效果好得多
- 硬编码阈值确保客观性

**劣势**：
- 需要手动设置评估标准
- Evaluator Agent 也需要调试

**元一视角**：
- **存续为体**：评估的目的是确保 Agent 持续产生价值
- **形式为用**：评估方法（Evaluator Agent、硬编码阈值）都是"形式"，可以调整
- **流动趋效**：Trace 分析帮助发现效率瓶颈
- **结构求稳**：评估标准相对稳定，但评估方法可以迭代

---

### 2. OpenDev 评估方法

**核心思想**：自动化测试 + 用户反馈

**评估流程**：
1. 每个 task 结束时，自动运行测试用例
2. 测试用例覆盖 happy path + edge case
3. 用户反馈作为补充（用户满意度调查）
4. Dashboard 实时显示评估结果

**优势**：
- 高度自动化，减少人工干预
- 实时反馈，快速迭代

**劣势**：
- 测试用例需要手动编写
- 用户反馈可能延迟

**元一视角**：
- **存续为体**：自动化测试确保 Agent 持续可用
- **形式为用**：测试用例、Dashboard 都是"形式"
- **流动趋效**：自动化评估加速反馈循环
- **接口预变**：Dashboard 可以对接各种评估数据源

---

### 3. LangChain 评估方法

**核心思想**：LangSmith Tracing + 自定义指标

**评估流程**：
1. 每个 chain 调用都被 LangSmith 记录
2. 自定义指标：准确性、延迟、成本、token 使用量
3. Trace 分析：找出失败模式、性能瓶颈
4. 手动 + 自动反馈闭环

**优势**：
- 细粒度可观测性（每个 chain 调用都有 Trace）
- 自定义指标灵活

**劣势**：
- LangSmith 需要额外学习成本
- 自定义指标需要手动定义

**元一视角**：
- **存续为体**：Trace 分析帮助发现系统健康问题
- **形式为用**：LangSmith、自定义指标都是"形式"
- **流动趋效**：Trace 分析帮助优化信息流动
- **结构求稳**：LangSmith 提供稳定的评估基础设施

---

### 4. Codex 评估方法

**核心思想**：单元测试 + 集成测试

**评估流程**：
1. 每个 PR 自动运行单元测试
2. 集成测试覆盖主要功能
3. 代码覆盖率作为质量指标
4. 手动代码审查

**优势**：
- 与现有软件开发流程无缝集成
- 代码覆盖率提供量化指标

**劣势**：
- 测试需要手动编写
- 代码覆盖率不等于功能正确性

**元一视角**：
- **存续为体**：测试确保代码质量，支撑系统存续
- **形式为用**：单元测试、集成测试都是"形式"
- **流动趋效**：自动化测试加速开发循环
- **结构求稳**：测试提供稳定的质量保障

## 评估方法对比总结

| 框架 | 最佳场景 | 关键优势 | 关键劣势 |
|------|----------|----------|----------|
| **Anthropic** | 复杂任务、需要高质量输出 | Evaluator Agent 提供客观评估 | 需要手动设置评估标准 |
| **OpenDev** | 快速迭代、自动化需求高 | 高度自动化，实时反馈 | 测试用例需要手动编写 |
| **LangChain** | 需要细粒度可观测性 | LangSmith Traces 提供详细信息 | LangSmith 需要额外学习成本 |
| **Codex** | 传统软件开发流程 | 与现有流程无缝集成 | 测试需要手动编写 |

---

## 元一思想评估

**最符合元一思想的评估方法**：Anthropic 的 Evaluator Agent 方法

**理由**：
1. **存续为体**：Evaluator Agent 确保 Agent 持续产生价值
2. **形式为用**：评估方法可以调整（Evaluator Agent、硬编码阈值）
3. **流动趋效**：Trace 分析帮助发现效率瓶颈
4. **结构求稳**：评估标准相对稳定

**最不符合的**：Codex 的单元测试方法

**理由**：
1. **过度形式化**：单元测试、代码覆盖率都是"形式"，可能损害"存续"（测试通过但功能不对）
2. **流动不畅**：手动代码审查可能导致信息流动受阻
3. **结构过于稳定**：测试标准可能过于僵化，不适应变化

---

## 启示

1. **评估方法应该多样化** — 不同场景需要不同的评估方法
2. **自动化很重要** — 但不能完全替代人工审查
3. **评估标准应该硬编码** — 但也要允许迭代
4. **Trace 分析是关键** — 帮助发现系统健康问题
