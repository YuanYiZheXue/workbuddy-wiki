# 自我验证循环

（Build-Verify Loop / Self-Verification Loop）

## 定义

Agent 完成一部分工作后，主动验证其正确性，再继续下一步。这是对抗「Agent 对自己的作品过度自信」的核心手段。

## 失败模式

> Agent 倾向于对自己的作品自信地赞美，即使对人类观察者来说质量明显平庸。

尤其在主观任务（如设计）中，没有可验证的正确性检查，Agent 的自我评估不可靠。

## 解决方案：分离生成与评估

| 角色 | 职责 | 对应实现 |
|------|------|----------|
| Generator | 生成方案 / 写代码 | Coding Agent / Builder Agent |
| Evaluator | 验证、找 bug、给出具体反馈 | QA Agent / Evaluator Agent |

> 让评估者变得「怀疑」，比让生成器批判自己的作品效果好得多。

## 具体实现方式

### Anthropic 方案（Generator + Evaluator）
- Evaluator 使用 Playwright MCP 直接与实时页面交互（不是评静态截图）
- 每条评估标准有硬性阈值，任一条不达标则 sprint 失败
- Sprint Contract：generator 和 evaluator 在写代码前先协商「完成」的定义

### LangChain 方案（Build & Self-Verify Prompting）
在 system prompt 里注入 4 步工作流：
1. **Planning & Discovery**：读任务、扫描代码库、制定计划
2. **Build**：以实现验证为目标写代码，写测试（happy path + edge case）
3. **Verify**：运行测试、读完整输出、对比原始需求（不是对比自己的代码）
4. **Fix**：分析错误、重读原始规格、修复问题

**PreCompletionChecklistMiddleware**：在 Agent 退出前拦截，强制跑一遍验证清单（类似 Ralph Wiggum Loop 的验证版）。

## 与 Wiki 方法的映射

| 自我验证循环 | Wiki 方法对应 |
|--------------|--------------|
| Evaluator 验证每个 sprint | Lint 工作流：定期健康检查 |
| Trace 分析找出失败模式 | 定期回顾 log.md，发现矛盾和过时信息 |
| 评估者给出具体可行动的反馈 | Ingest 后更新 index，确保干净状态 |
