# 工程技术：在智能体优先的世界中利用 Codex

> OpenAI 团队用 Codex 从零构建一款产品，全程无人手写代码，约 100 万行代码、1500 个 PR，仅用 3 名工程师。

## 核心观点

### 1. 人类角色重新定义
- 工程师的工作重点转向**系统、架构和杠杆**
- 人类不直接写代码，而是通过提示与 Codex 交互
- "不手动编写代码"是核心理念
- 人类介入的方式是追问："还需要什么能力，如何让这个能力对智能体清晰可读且可执行？"

### 2. Ralph Wiggum 循环（自我修复循环）
- Codex 打开 PR → 自我审查 → 响应反馈 → 循环直到所有审查通过
- 人类可以审核 PR，但非必须；逐渐转为智能体对智能体的审核方式
- 引用：[Ralph Wiggum 循环](https://ghuntley.com/loop/)

### 3. 提高应用对智能体的可读性
- 让 UI、日志、指标对 Codex **直接可读**
- 接入 Chrome DevTools 协议，Codex 能复现错误、验证修复
- 接入完整可观测性堆栈（Logs/Metrics/Traces），用 LogQL/PromQL 查询
- 单次 Codex 运行可持续工作 6+ 小时（人类睡眠时间）

### 4. 代码仓库作为记录系统
- **给 Codex 的是地图，不是 1000 页的说明书**
- 失败尝试：一个巨大的 `AGENTS.md`（上下文被挤占、指导过多反而无效、立即腐烂、难以核实）
- 成功做法：`AGENTS.md` 作为**内容目录**（约 100 行），指向 `docs/` 目录中的结构化知识
- 渐进式披露：智能体从小而稳定的切入点开始，按需深入

### 5. 目标是智能体可读性
- Codex 看不到的东西 = 不存在（Google Docs、Slack、人脑中的知识均不可访问）
- 所有知识必须以 Markdown 形式编码到代码仓库中
- 偏好"枯燥"技术：可组合、API 稳定、训练集中表现好
- 有时重新实现部分功能比引入不透明的上游依赖更便宜

### 6. 规范架构与品味
- 强制执行**不变量**，而非微观管理实施过程
- 分层领域架构：Types → Config → Repo → Service → Runtime → UI
- 横切关注点（认证、遥测等）通过单一显式接口（Providers）进入
- 自定义 linter 强制执行规则，错误信息中注入修复指令
- 生成的代码不必符合人类风格偏好，只要对**未来智能体运行**清晰可读即可

### 7. 吞吐量改变合并理念
- 尽量减少阻塞合并门，PR 生命周期短
- 测试偶发失败通过重跑解决，而非无限期阻碍进展
- 纠错成本低，等待成本高

### 8. 熵与垃圾回收
- Codex 会复现代码仓库中已存在的（甚至不理想的）模式 → 导致漂移
- 早期：人工每周五花 20% 时间清理"AI 残渣" → 不可扩展
- 改进：将"黄金原则"编码到代码仓库，建立循环清理流程
- 定期运行后台 Codex 任务，扫描偏差、更新质量等级、发起重构 PR
- 类似垃圾回收：小额高频偿还技术债务 >> 一次性痛苦解决

## 对 Wiki 方法的启示

1. **Ralph Wiggum 循环** → Ingest 流程中可以加入"自我验证"步骤
2. **渐进式披露** → wiki 的概念页/实体页应该是分层的，摘要页不应该试图包含所有细节
3. **智能体可读性** → 写 wiki 页面时要考虑"WorkBuddy 能读懂吗"，而不是"人类能读懂吗"
4. **垃圾回收** → Lint 流程不仅要查矛盾，还要主动清理过时内容
5. **代码仓库即记录系统** → Wiki 本身就是记录系统，所有知识必须在文件中，不能只在聊天历史中
