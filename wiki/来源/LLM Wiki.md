# LLM Wiki

（本页面由 AI 根据原始资料自动生成，请人工审核）

## 核心论点

不是让 LLM 在查询时重新发现知识，而是让 LLM **增量构建并维护一个持久化的 Wiki**——一个结构化的、相互链接的 Markdown 文件集合，位于你和原始资料之间。

**关键区别**：Wiki 是一个持久的、具有复合效应的产物。知识是被编译一次然后*保持最新*，而不是每次查询都重新推导。

## 关键概念

### 三层架构

#### 1. 原始资料
- 你精心策划的源文档集合
- 文章、论文、图片、数据文件
- 不可变——LLM 从中读取但从不修改
- 这是你的真理来源

#### 2. Wiki
- 由 LLM 生成的 Markdown 文件目录
- 摘要、实体页面、概念页面、对比、概览、综合结论
- LLM 完全拥有这一层
- 它创建页面、在新资料到达时更新它们、维护交叉引用，并保持一切一致
- 你阅读它；LLM 写它

#### 3. Schema
- 一个文档（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md）
- 告诉 LLM Wiki 是如何结构的、约定是什么、以及在摄取资料、回答问题或维护 Wiki 时要遵循什么工作流
- 这是关键的配置文件
- 你和 LLM 随着时间的推移共同演进这个文件

### 三种操作

#### 1. 摄取（Ingest）
- 你将一个新的资料来源放入原始集合，并告诉 LLM 处理它
- 示例流程：
  1. LLM 阅读资料来源
  2. 与你讨论关键要点
  3. 在 Wiki 中写一个摘要页面
  4. 更新索引
  5. 更新 Wiki 中各相关的实体和概念页面
  6. 追加一个条目到日志
- 一个单一的资料来源可能触及 10-15 个 Wiki 页面
- 你可以一次摄取一个资料并保持参与，也可以批量摄取许多资料

#### 2. 查询（Query）
- 你针对 Wiki 提问
- LLM 搜索相关页面，阅读它们，并综合一个带有引用的答案
- 根据问题的不同，答案可以采取不同的形式——一个 Markdown 页面、一个对比表格、一套幻灯片（Marp）、一张图表（matplotlib）、一个画布
- **重要**：好的答案可以作为新页面归档回 Wiki
- 你要求的对比、分析、联系——这些都是有价值的，不应该消失在聊天历史中

#### 3. 整理（Lint）
- 定期要求 LLM 对 Wiki 进行健康检查
- 查找：
  - 页面之间的矛盾
  - 已被更新的资料来源取代的过时主张
  - 没有入站链接的孤儿页面
  - 被提到但缺少自己页面的重要概念
  - 缺失的交叉引用
  - 可以通过网络搜索填补的数据空白
- LLM 擅长建议新的调查问题和寻找新的资料来源
- 这使得 Wiki 在增长时保持健康发展

### 索引和日志

#### index.md
- 面向内容的目录
- Wiki 中一切内容的目录——每个页面列出一个链接、一行摘要，以及可选的元数据如日期或来源数量
- 按类别组织（实体、概念、来源等）
- LLM 在每次摄取时更新它
- 在回答查询时，LLM 首先阅读索引以找到相关页面，然后深入其中
- 在中等规模（约 100 个来源，约数百个页面）下效果出奇地好，并避免了对基于嵌入的 RAG 基础设施的需求

#### log.md
- 按时间顺序排列
- 一个仅追加的记录，记录发生了什么以及何时发生——摄取、查询、整理操作
- 一个有用的提示：如果每个条目都以一致的前缀开头（例如 `## [2026-04-02] ingest | Article Title`），那么日志可以用简单的 unix 工具解析——`grep "^## \[" log.md | tail -5` 给你最后 5 个条目
- 日志给你 Wiki 演进的时间线，并帮助 LLM 了解最近做了什么

## 与 Wiki 主题的联系

### 与「元一思想」的关系

LLM Wiki 理念体现了元一思想的多个原则：

1. **存续为体**：Wiki 的价值在于被使用，不是为了完美而存在
2. **形式为用**：Wiki 格式、Schema 约定都是"形式"，可以随时调整
3. **流动趋效**：页面间链接要通畅，信息查找成本要低
4. **结构求稳**：Schema 提供稳定结构，但内容可以演进

### 与「WorkBuddy Wiki Schema」的关系

我们现在正在实现的 `workbuddy-wiki-schema.md` 就是 LLM Wiki 理念的具体化：
- 三层架构 → 原始资料（`raw/`）、Wiki（`wiki/`）、Schema（`workbuddy-wiki-schema.md`）
- 三种操作 → 摄取（Ingest）、查询（Query）、整理（Lint）
- 索引和日志 → `wiki/index.md`、`wiki/log.md`

**区别**：
- LLM Wiki 理念是通用的，适用于任何 LLM Agent
- WorkBuddy Wiki Schema 是具体的，针对 WorkBuddy 优化
