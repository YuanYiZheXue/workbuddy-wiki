# 子空间迭代管理规范

> 版本：v1.0.0  
> 创建时间：2026-04-27  
> 作者：元一 📘

---

## 核心原则

**子空间的内容不要合并到主仓库。**

每个子空间 agent（哲哲/数数/算算/金金）独立迭代，主仓库只记录他们的迭代历史，不合并他们的代码或文档。

## 迭代记录内容

每个 agent 的迭代记录文件 (`哲哲.md` / `数数.md` / `算算.md` / `金金.md`) 应包含：

### 1. 基本信息

```markdown
> Agent：哲哲 📕  
> 负责：哲学知识库  
> 分支：`kb/philosophy`  
> 工作空间：`d:/Obsidian_KN/哲学思想`
```

### 2. 提交记录

| 提交哈希 | 提交信息 | 说明 |
|---------|---------|------|
| `hash` | `message` | description |

### 3. 迭代内容

描述本次迭代的核心内容，例如：
- 新增了哪些文件/功能
- 修改了哪些文件/功能
- 采用了什么新方法/新思路

### 4. 问题清单（如有）

描述遇到的问题和修复方案，例如：
- 文件名笔误
- 提交了不该提交的文件
- 双远程同步失败

### 5. 主仓库决策

描述主仓库是否采纳子空间的迭代，例如：
- ❌ 不合并（子空间内容独立）
- ⏳ 待定（需评估是否纳入主仓库）
- ✅ 采纳（部分采纳，说明采纳了什么）

## 工作流程

### 子空间 agent 提交迭代

1. 子空间 agent 在自己的分支上提交代码/文档
2. 子空间 agent 通知主仓库 agent（元一）有新的迭代
3. 元一查看子空间分支的提交记录
4. 元一将迭代内容记录到对应的 `.workbuddy/iterations/X.md` 文件

### 主仓库审查迭代

1. 元一查看子空间的提交记录和迭代内容
2. 元一评估是否采纳子空间的迭代（全部/部分/不采纳）
3. 元一在迭代记录文件中记录决策
4. 如果采纳，元一在主仓库中实现相同的功能（而不是合并子空间的代码）

### 子空间修复问题

1. 子空间 agent 发现自己的问题（或元一指出问题）
2. 子空间 agent 在自己的分支上修复问题
3. 子空间 agent 通知元一问题已修复
4. 元一更新迭代记录文件中的「修复进度」

## 注意事项

### 1. 子空间内容不合并到主仓库

- 子空间 agent 的迭代是独立的，不应该合并到主仓库
- 如果主仓库需要类似功能，应该在主仓库中重新实现（而不是合并子空间的代码）

### 2. 迭代记录只记录，不执行

- `.workbuddy/iterations/*.md` 只记录迭代历史，不执行任何操作
- 如果需要执行（例如采纳子空间的某个想法），应该在主仓库中创建新的文件/提交

### 3. 定期审查迭代记录

- 元一应该定期审查子空间的迭代记录
- 如果发现有价值的想法，可以考虑采纳到主仓库

### 4. 保护子空间的独立性

- 不要修改子空间的迭代记录文件（除非是元一自己记录）
- 子空间 agent 应该自己维护自己的迭代记录（或者通知元一更新）
