## 核心理念

大多数人与 LLM 和文档交互的体验类似于 RAG：你上传一批文件，LLM 在查询时检索相关片段，然后生成答案。这能工作，但 LLM 对每个问题都是从零开始重新发现知识，没有积累。

这里的理念不同。不是在查询时仅从原始文档中检索，而是让 WorkBuddy **增量构建并维护一个持久化的 Wiki**——一个结构化的、相互链接的 Markdown 文件集合，位于你和原始资料之间。当你添加一个新的资料来源时，WorkBuddy 不只是为其建立索引，而是阅读它、提取关键信息，并将其整合到现有 Wiki 中。知识是被编译一次然后*保持最新*，而不是每次查询都重新推导。

**Wiki 是一个持久的、具有复合效应的产物。** 交叉引用已经存在。矛盾已经被标记。综合结论已经反映了你阅读过的所有内容。每添加一份资料、每提出一个问题，Wiki 都会变得更加丰富。

你从不（或很少）自己写 Wiki——WorkBuddy 编写并维护其中的所有内容。你负责资料来源、探索和方向、提出正确的问题。WorkBuddy 做所有的苦力活。在实践中，把 WorkBuddy 放在一边，Obsidian 放在另一边。WorkBuddy 根据你们的对话进行编辑，你实时浏览结果。

## 架构

有三层：

**原始资料**——你精心策划的源文档集合。文章、论文、图片、数据文件。这些是不可变的——WorkBuddy 从中读取但从不修改它们。

**Wiki**——一个由 WorkBuddy 生成的 Markdown 文件目录。摘要、实体页面、概念页面、思想页面、对比、概览。WorkBuddy 完全拥有这一层。它创建页面、在新资料到达时更新它们、维护交叉引用。你阅读它；WorkBuddy 写它。

Wiki 的页面类型分三层：
- **组织层**：章首页（Chapter Overview）— 每章的顶层地图
- **智慧层**：思想页（Thinking）— 数学哲学和方法论（"怎么想"）
- **知识层**：概念页（Concept）、实体页（Entity）、对比页（Comparison）
- **索引层**：来源页（Source）、各类 index

> **关键洞察（来自 Agent Harness 理论）**：文件系统是天然的协作面（collaboration surface）。多个 Agent 和人类可以通过共享文件协调工作。Wiki 方法正是利用这一点——Wiki 是 WorkBuddy 与你之间的协作面，所有知识持久化在文件系统中，不依赖上下文窗口。

**Schema**——本文件（以及 `SOUL.md`、`IDENTITY.md`），告诉 WorkBuddy Wiki 是如何结构的、要遵循什么工作流。这是关键的配置文件。你和 WorkBuddy 随着时间的推移共同演进这个文件。

## 操作

### 每次会话启动（参考：Anthropic 的 "get up to speed" 实践）

WorkBuddy 在每次会话开始时，应先了解当前状态，再开始工作：

1. 读取 `wiki/index.md` 了解当前 Wiki 规模
2. 读取 `wiki/log/index.md` 了解近期操作（点击日期链接查看详情）
3. 读取 `.workbuddy/memory/MEMORY.md` 获取跨会话重要事实
4. 读取本文件（`workbuddy-wiki-schema.md`）确认当前遵循的约定

这对应 Anthropic 在长运行 Agent 研究中提出的做法：Agent 每次拿到新上下文窗口时，首先要"找到方向"（get its bearings），而不是盲目开始工作。

### 会话管理

WorkBuddy 历史对话最多保存 **100 个**。超过后最早的历史会被丢弃。

**影响**：
- 每个会话不要超过 ~30 轮对话（预留 buffer）
- 长任务应拆分为多个会话
- 会话结束时，关键信息必须写入持久化文件（log、memory、index）

**建议工作流**：
1. 每 ~3 个对话开启新对话
2. 新对话开头：`新对话开始。请先读取 wiki/log/index.md 和最新的 wiki/log/YYYY-MM-DD.md，了解上次工作进度，然后继续。`
3. 会话结束前，确保所有重要信息已写入 `wiki/log/YYYY-MM-DD.md` 和 `.workbuddy/memory/YYYY-MM-DD.md`

> **注意**：此约束与 Agent 配置相关。其他知识库的 Agent 应根据自己的历史对话保存上限调整策略。

### Ingest（摄取）

**核心原则：一次只处理一篇资料（增量原则）。** 对应 Anthropic 研究中"一次只做一个功能"的经验。

#### 资料来源类型与处理方式

| 来源类型 | 处理方式 |
|----------|----------|
| 本地文件（PDF/MD/TXT） | 直接读取 `素材/` 目录中的文件 |
| 网页 URL | 用 Defuddle 提取干净 Markdown：`defuddle parse <url> --md -o 素材/<标题>.md`，再按本地文件流程处理 |
| 用户直接提供文本 | 由用户粘贴内容，WorkBuddy 写入 `素材/` 后按本地文件处理 |

> **Defuddle 优势**：去除网页导航、广告、侧边栏等噪音，只保留正文 Markdown，减少 token 消耗。比 WebFetch 更适合作为 Ingest 的资料预处理步骤。

你将一个新的资料来源放入 `素材/`，并告诉 WorkBuddy 处理它。示例流程：

1. WorkBuddy 阅读原始资料
2. 与你讨论关键要点和侧重点
3. 创建 `wiki/来源/YYYY-MM-DD 标题.md` 摘要页
4. 更新相关的实体页和概念页（遵循自下而上原则）
5. **提炼数学思想，创建思想页（参见「数学思想」章节）**
6. 更新 `wiki/index.md`（Stats + 条目）
7. 追加 `wiki/log/index.md`（或对应日期文件 `wiki/log/YYYY-MM-DD.md`）
8. 追加 `.workbuddy/memory/YYYY-MM-DD.md`

> **思想页创建时机**：在 Ingest 过程中，当发现以下情况时，应在步骤5创建思想页：
> - 资料作者有明显的哲学选择（如 Rudin 不构造实数，直接给公理）
> - 出现反复使用的证明策略（如：先证特殊再推广、反证法构造矛盾）
> - 抽象跃迁有明显的层次路径（如：ℝ → 度量空间 → 拓扑空间）
> - 反例有设计意图（如：康托集同时展示"不可数"和"测度为零"）

**结束时确保干净状态**：index、log、memory 都已更新，Wiki 处于一致状态。这对应 Anthropic 研究中"leave a clean state"的实践。

> **Ralph Wiggum 循环（自我验证）**：汲取自 OpenAI Codex 实践。Ingest 结束后，WorkBuddy 应自我检查：index 更新了吗？log 追加了吗？交叉引用正确吗？对应"结束时确保干净状态"——不是人类来检查，而是 WorkBuddy 自己验证自己的工作。详见 [[概念/Ralph Wiggum 循环]]。

> **渐进式披露（Progressive Disclosure）**：对应 LangChain 的 Skills 理念。WorkBuddy 不需要在每次会话开始时加载所有工具和 Skills；只在需要时调用 `use_skill`。这保护了上下文窗口，避免 Context Rot。Ingest 流程中如需特殊能力（如处理 PDF、操作 Excel），再按需加载对应 Skill。详见 [[概念/渐进式披露]]。

### Query（查询）

你针对 Wiki 提问。WorkBuddy 搜索相关页面，阅读它们，并综合一个带有引用的答案。**好的答案可以作为新页面归档回 Wiki**——你要求的对比、一个分析、一个你发现的联系——这些都是有价值的，不应该消失在聊天历史中。

### Lint（健康检查 + 垃圾回收）

定期要求 WorkBuddy 对 Wiki 进行健康检查。具体检查清单：

**必检项目：**
1. **悬空内部链接** — 扫描所有 `[[页面名]]`，确认目标页面存在；不存在的链接要么删除，要么创建对应页面（遵循自下而上原则）
2. **双向链接完整性** — 用 `obsidian backlinks file="页面名"` 验证每个页面的反向链接是否完整（汲取自 `obsidian-cli` skill）
3. **Frontmatter 拼写** — 检查 `sources:`、`tags:`、`type:` 等字段名拼写（常见错误：`sourses`、`tgas`、`tpye`）
4. **命名一致性** — 页面文件名与正文中提及的名称是否一致（如 `Ralph Wiggum 循环` vs `Ralph Loop`，应在正文里加别名说明）
5. **index.md 简介质量** — 每个条目的一句话简介是否有实质内容，不能只是重复页面标题
6. **孤儿页面** — 没有任何其他页面链接到它的页面
7. **过时主张** — 对比 `素材/` 原始资料和 `wiki/来源/` 摘要页，标记不再准确的描述
8. **思想页覆盖率** — 已创建的概念页中，是否有对应的思想页？（如：有 `概念/紧集`，应检查是否有 `思想/紧性的威力`）

> **`obsidian-cli` 加速 Lint**：如果 Obsidian 正在运行，可用 `obsidian backlinks file="Foo.md"` 直接获取反向链接列表，不需要 AI 自己扫描整个 wiki（适合大型 wiki）。依赖 Obsidian 运行，非 Obsidian 环境回退到 Python 脚本扫描。

**垃圾回收（Garbage Collection）**
> 汲取自 OpenAI Codex 实践。Codex 会复现代码仓库中已存在的模式（包括不理想模式），导致漂移。Wiki 也一样——过时的表述、不再准确的摘要会积累。

具体做法：
- 定期扫描 Wiki 中不再反映真实情况的页面
- 小额高频清理：每次 Lint 处理 1-2 个页面，而不是一次性大扫除
- 记录清理决策到 `wiki/log/index.md`（或对应日期文件 `wiki/log/YYYY-MM-DD.md`）
- 对应"技术债务就像高息贷款"理念：不断小额偿还 >> 一次性痛苦解决

## 索引和日志

两类文件帮助 WorkBuddy（和你）在 Wiki 增长时导航：

**wiki/index.md**——内容目录，按类别组织（来源、章首页、思想、概念、实体、对比）。WorkBuddy 在每次摄取时更新它。

索引结构：
| 类别 | 目录 | 说明 |
|----------|----------|----------|
| 来源（Sources） | `wiki/来源/` | 每篇原始资料的摘要页 |
| 章首页（Chapter Overviews） | `wiki/章首页/` | 每章的顶层组织页 |
| 思想（Thinking） | `wiki/思想/` | 数学哲学和方法论 |
| 概念（Concepts） | `wiki/概念/` | 核心概念页 |
| 实体（Entities） | `wiki/实体/` | 数学家、定理、书籍 |
| 对比（Comparisons） | `wiki/对比/` | 跨资料观点对比 |

**wiki/log/**——按日期组织的操作日志目录：
- `wiki/log/index.md`：总索引，按日期列出所有日志条目
- `wiki/log/YYYY-MM-DD.md`：当天的所有操作记录
- 每次 ingest、query、lint 都追加到对应日期的文件中
- 方便回顾 Wiki 的演进历史，且单文件不会无限增长

## 利用 WorkBuddy 的特有功能

### Memory 集成

WorkBuddy 有结构化的记忆机制（`.workbuddy/memory/`），可以在会话之间持久化上下文：

- 每次 ingest 后，记录关键信息到 `.workbuddy/memory/YYYY-MM-DD.md`
- 重要的跨会话事实，更新到 `.workbuddy/memory/MEMORY.md`
- 这样即使开启新会话，WorkBuddy 也能记住之前构建的 Wiki 内容

### Automation 集成

可以利用 WorkBuddy 的 automation 功能设置定期任务：

- 每周自动 lint Wiki
- 每日结束时建议下一步行动
- 提醒有待处理的原始资料

## 动态视图（Bases）

> 汲取自 Obsidian Bases 功能。**静态索引需要手动维护，Bases 让索引自动生成。**

### 理念

传统的 `wiki/index.md` 是静态列表，每次 ingest 后都要手动更新。Obsidian Bases（`.base` 文件）基于 frontmatter 属性动态聚合、排序、过滤，不需要手动维护索引。

**符合 wiki 思想**：索引本身应该是可查询的，而不是手工维护的列表。

### 何时创建 Base 文件

| 场景 | 动作 |
|------|------|
| 某类页面 ≥ 5 个，且都有结构化 frontmatter | 创建对应 `.base` 文件 |
| 需要在 Obsidian 中以表格/卡片视图浏览某类页面 | 创建 `.base` 文件 |
| 页面数量少（< 5）或 frontmatter 不规范 | 暂不创建，先用静态索引 |

### Base 文件规范

- **存放位置**：与对应 `index.md` 同级
  - `wiki/概念/概念列表.base`
  - `wiki/来源/来源列表.base`
  - `wiki/实体/实体列表.base`
- **嵌入方式**：在 `index.md` 末尾加入 `![[概念列表.base]]`，Obsidian 用户看到动态视图，非 Obsidian 用户看到静态 fallback 表格
- **视图类型选择**：
  - `table`：需要对比多个属性时（概念页、实体页）
  - `cards`：以视觉卡片浏览时（来源页，有封面图片）
  - `list`：简单列表即可时

### 最小示例

```yaml
# wiki/概念/概念列表.base
filters:
  and:
    - 'file.inFolder("wiki/概念")'
    - 'file.ext == "md"'

formulas:
  updated_date: 'file.mtime.format("YYYY-MM-DD")'

properties:
  title:
    displayName: 名称
  formula.updated_date:
    displayName: 更新日期

views:
  - type: table
    name: 所有概念
    order:
      - file.name
      - title
      - formula.updated_date
    groupBy:
      property: file.name
      direction: ASC
```

### 与非 Obsidian 环境的兼容

`.base` 文件只有 Obsidian 能渲染。为非 Obsidian 用户：
- `index.md` 保留静态表格作为 fallback
- 在 `index.md` 顶部加注释：`<!-- Obsidian 用户可查看对应的 .base 文件获取动态视图 -->`

## 为什么这有效

维护知识库的乏味部分不是阅读或思考——而是簿记。更新交叉引用、保持摘要最新、注意新数据何时与旧主张矛盾。人类放弃 Wiki 是因为维护负担增长得比价值快。WorkBuddy 不会感到无聊，不会忘记更新交叉引用，并且可以在一次传递中触及多个文件。Wiki 保持维护是因为维护成本接近零。

## 顶层设计：元一思想

> 汲取自用户原创思想体系「元一思想」：**Wiki 本身是一个元一系统，四原则指导所有构建决策。**

### 四原则在 Wiki 构建中的具体体现

| 原则 | Wiki 构建中的含义 | 具体操作 |
|------|-------------------|----------|
| **存续为体，形式为用** | Wiki 的价值在于被使用，不是为了完美而存在 | 不过度设计 Schema；页面格式服务于可读性，而非反之 |
| **流动趋效，均衡为度** | 页面间链接要通畅，信息查找成本要低 | 用 `[[页面名]]` 做交叉引用；避免信息孤岛；定期 Lint 检查链接 |
| **意义生于博弈，固于认同** | Wiki 的内容优先级由实际使用需求驱动 | 自下而上原则：先有素材，再建概念页；不凭空创建页面 |
| **结构求稳，接口预变** | Schema 既稳定又可扩展 | 版本化 Schema（v1/v2/v3）；新需求先想「能否通过接口扩展」，而非推翻重来 |

### 四极飞轮（Wiki 健康度诊断模型）

```
            流动趋效，均衡为度
                  ↗️   ↖️
  存续为体 ← 四极飞轮 → 意义生于博弈
   (轴心)        ↖️   ↗️      固于认同
             结构求稳，接口预变
```

定期用此模型诊断 Wiki 状态（见 [[概念/元一思想]] 中的健康度自检清单）。

### 接口决策规则（新需求来时如何判断？）

> 对应第四原则「结构求稳，接口预变」。当有新需求时，按此流程判断：

**决策树：**

```
新需求来了
  ├─ 是否是已有概念的细化？
  │   └─ 是 → 扩展现有页面内容（接口预变）✅
  ├─ 是否是全新概念/实体？
  │   └─ 是 → 按自下而上原则：先有素材，再建页 ✅
  └─ 是否需要修改 Schema 核心流程？
      └─ 先问：「能否通过接口扩展？」
          ├─ 能 → 新增约定/示例，不修改核心流程 ✅
          └─ 不能 → 只有以下情况才修改结构：
              ├─ 现有流程被证明有系统性缺陷（≥3次相同错误）
              ├─ 新需求与现有结构根本冲突
              └─ 否则：在 log.md 中记录，不修改 Schema
```

**判断清单（修改结构前必问）：**

1. 这个需求是**一次性**的还是**系统性**的？
   - 一次性 → 在 `log.md` 中记录，不修改 Schema
   - 系统性 → 进入下一问
2. 系统性的需求能否通过**扩展接口**解决（新增章节/细则）？
   - 能 → 扩展，不修改核心结构 ✅
   - 不能 → 才修改结构（Schema 版本号 +1）

> **实例**：「自下而上原则」是结构（稳）；「新概念页的 frontmatter 模板」是接口（可预变）。新需求来时，先想能否通过更新模板（接口）解决，而非推翻自下而上原则（结构）。

## 数学思想（Mathematical Thinking）

> 汲取自 Rudin《数学分析原理》第1-4章。**Wiki 不仅记录"是什么"（概念页），还应该记录"怎么想"（数学思想）。前者是知识，后者是智慧。**

### 理念

数学思想是从资料中提炼的**高层数学哲学和方法论**，不同于具体概念定义。

包括：
- **证明策略**（如：先证特殊情况，再推广；用反证法构造矛盾）
- **公理化的选择**（如：Rudin 不构造实数，直接给公理——存在性只需简要说明）
- **抽象跃迁的路径**（如：具体到抽象：ℝ → 度量空间 → 拓扑概念）
- **反例的设计哲学**（如：Cantor 集为什么是"完美"反例）
- **结构的层次关系**（如：紧性 → 一切好事情发生）

### 何时创建思想页

| 触发条件 | 动作 |
|----------|------|------|
| 从资料中发现反复出现的证明策略 | 创建 `wiki/思想/XXX.md` |
| 资料作者有明显的哲学选择（公理 vs 构造、具体 vs 抽象） | 创建思想页记录这种选择 |
| 某个思想可以跨章节、跨资料应用 | 创建思想页，并链接到相关概念页 |
| 只是单个概念的定义 | 不创建思想页，写在概念页的"深入"部分 |

### 思想页格式规范

```markdown
---
title: "思想名称"
type: 思想
source: ["资料1", "资料2"]
related: [[概念/XXX]], [[概念/YYY]]
---

# 思想名称

> 一句话概括这种思想的核心。

## 核心内容

## 在资料中的体现

（引用具体章节、定理、证明技巧）

## 与其他思想的关系
```

### 已识别的数学思想（来源：Rudin 第1-4章）

| 思想 | 体现章节 | 说明 |
|------|----------|------|
| 公理化方法 | 第1章 | 不构造实数，直接给公理；存在性简要说明即可 |
| 具体到抽象 | 第2章 | ℝ → 度量空间 → 拓扑概念，逐层抽象 |
| 等价刻画 | 第4章 | 同一概念多角度定义（ε-δ ≡ 序列 ≡ 开集原像） |
| 反例的力量 | 第2/4章 | Cantor集、Dirichlet函数、有界无最值函数 |
| 紧性的威力 | 第2/4章 | 紧集 → 最值定理、一致连续、连续映射保紧 |
| 一致 vs 逐点 | 第4章 | 一致连续比连续强；δ 与 p 无关是关键 |
| 确界驱动收敛 | 第1/3章 | 单调有界→收敛，本质是实数完备性 |

### 与现有页面类型的关系

```
概念页（What） ← 基础
    ↑
思想页（How & Why） ← 上层
    ↑
章首页（Chapter Map） ← 组织层
```

- **概念页** 回答"它是什么"（定义、定理、例子）
- **思想页** 回答"为什么这样想"、"这反映了什么深层哲学"
- **章首页** 提供章节的概念地图和学习路线

## Wiki 写作规范

> 汲取自 OpenAI Codex 工程实践：**Codex 看不到的东西 = 不存在。**
> 汲取自 `obsidian-markdown` skill：用 Obsidian 特有语法提升智能体可读性。

### 智能体可读性（Agent Readability）

Wiki 的读者是 WorkBuddy，不是人类。写每个页面时都要问：**WorkBuddy 能直接从这个页面推理出完整含义吗？**

#### 基础实践
- 用 `[[页面名]]` 做交叉引用，不要用"见上文"或"同上"
- Frontmatter 用结构化字段，不要用自由文本描述
- 避免"如前所述"——智能体无法回溯上下文，必须显式重写关键信息
- 图片用 `![[素材/图片名.png]]` 引用，确保路径对 WorkBuddy 可读（图片存放于 `素材/` 目录）

#### Obsidian 语法增强（汲取自 `obsidian-markdown`）

**Callouts（标注元信息）**

用 callout 标注页面中的元信息，比普通段落更有结构：

```markdown
> [!info] 关键定义
> 这里是核心定义，智能体读取时能被结构识别。

> [!warning] 争议点
> 此概念存在不同解释，目前采用 XX 学派观点。

> [!todo] 待验证
> 此主张需要进一步资料来源确认。
```

推荐类型：`note`、`info`、`warning`、`tip`、`bug`、`question`、`example`

**Block References（精确链接到段落）**

当需要引用页面内的某个具体论点时，用 block ID 而不是整个页面：

```markdown
这是一个重要结论，需要被单独引用。^important-conclusion
```

其他页面用 `[[页面名#^important-conclusion]]` 精确链接。

**Comments（AI 内部笔记）**

用 `%% comment %%` 在页面中留 AI 笔记，人类在 Obsidian 阅读视图里看不到，但 AI 下次读取时能获取上下文：

```markdown
%% WorkBuddy 笔记：此页面与 [[概念/XXX]] 存在部分重叠，下次 Lint 时需合并 %%`
```

**高亮**

用 `==高亮文本==` 标记关键信息，帮助智能体快速定位重点。

### 本文档是地图，不是百科全书（渐进式披露）

`workbuddy-wiki-schema.md` 的定位是**内容目录**，不是详细操作手册。

- 本文档保持精简（目标 100-150 行），只描述高层理念和关键约定
- 详细规范应在各自的参考文档中
- WorkBuddy 从小而稳定的切入点开始，按需深入

> 失败案例：一个巨大的配置文件会挤掉任务相关的上下文，且会立即腐烂。

## 注意

本文档是有意抽象的。它描述的是理念，而不是具体的实现。确切的目录结构、页面格式、工具——所有这些都取决于你的领域、你的偏好和你的工作习惯。

**以上一切都是可选和模块化的——选择有用的，忽略没用的。** 例如：
- 你的资料来源可能只是纯文本的，所以你根本不需要图片处理
- 你的 Wiki 可能足够小，索引文件就是你所需要的一切，不需要搜索引擎
- 你可能不关心幻灯片，只想要 Markdown 页面
- 你可能想要一套完全不同的输出格式

使用它的正确方法是把本文档分享给 WorkBuddy，共同工作以实例化一个适合你需求的版本。本文档的唯一工作是传达模式。WorkBuddy 可以弄清楚其余的事情。

## 迭代改进本 Schema

> 汲取自 Anthropic 的 Harness 设计实践：**Harness 中的每个组件都编码了一个关于「模型不能独自做什么」的假设。随着模型改进，这些假设会过时——要定期审视、去掉不再承重（load-bearing）的部分。**

具体做法：

- 每处理完一批素材，回顾当前 Schema 的每一条规则：「如果模型升级了，这条还必要吗？」
- 如果某条规则已经由模型原生能力覆盖，删掉它，而不是保留「以防万一」
- 反之，如果发现新的失败模式，及时加进来
- Schema 不是静态文档，它应该随着模型和你的需求共同演进

> 这对应 Wiki 方法本身的核心精神：**复合效应**——Wiki 随每次摄取和每次反思而变得更丰富，Schema 也一样。

## 内容分发（Content Distribution）

> 汲取自 `📺 Bilibili Skill`、`OpenClaw Video Publisher`、`小红书助手`。——**先设计接口，等平台配置好后再实施**。

### 理念

Wiki 的内容需要被分发到不同平台（B 站、小红书、抖音等），并从平台反馈中汲取改进点。这是「意义生于博弈，固于认同」的实践——内容在传播中与读者碰撞，产生新的意义共识。

### 接口设计（扩展点）

内容分发是 **接口扩展**，不修改 Schema 主体。当各平台配置好后，按以下流程执行：

```
Wiki 内容 → 适配各平台格式 → 发布 → 收集反馈 → 汲取到 Wiki
```

### 各平台发布格式规范

| 平台 | 内容形式 | 格式要点 | 状态 |
|------|----------|----------|--------|
| B 站动态 | 短文（≤500 字） | 提炼核心观点 + 标签 + 链接 | 🟡 脚本待调试 |
| 小红书 | 图文笔记 | 标题 + 正文 + 标签 + 封面图 | 🔴 MCP 未配置 |
| 抖音/快手 | 视频 | 需先制作视频内容 | ⚪ 未开始 |
| 掘金/知乎 | 长文 | 已发布，维护更新 | ✅ 已完成 |

### 发布前检查清单

- [ ] 内容已提炼为适合平台的格式
- [ ] 标签/话题已配置
- [ ] 发布后链接已记录到 `wiki/发布/index.md`
- [ ] 评论监控已设置（feedback-to-wiki 循环）

### 反馈汲取流程

```
平台评论/私信 → 归类（bug/需求/疑问/认可）
       ↓
  记录到对应 Wiki 页面（用 Comments 语法：%% 反馈：xxx %%）
       ↓
  Lint 时检查 Comments，更新页面内容
       ↓
  意义共识被更新，Wiki 变强
```

### 阻塞处理记录

| 平台 | 阻塞原因 | 绕开策略 | 后续路径 |
|------|----------|----------|----------|
| B 站 | `send_dynamic` API 返回 `-400` | 脚本已创建，暂不强制调试 | 用浏览器抓包对照 `bilibili-api-python` 源码 |
| 小红书 | MCP 服务未配置（`mcp.json` 为空） | 先设计流程规范 | 运行 `setup-xhs-mcp` 部署服务 |

> **流动趋效原则应用**：当某平台发布阻塞时，先完成其他平台或设计流程规范，不在一个阻塞点上消耗过多时间。

## Schema 版本化管理（Git-Based Distribution）

> 对应第四原则「结构求稳，接口预变」。Schema 独立版本化，通过 Git 实现跨工作区、跨贡献者的双向同步。

### 背景与问题

当 Wiki 构建系统需要部署到多个工作区、或多个贡献者协作时，面临四个问题：

| 问题 | 说明 |
|------|------|
| 跨工作区同步 | 其他工作区如何获取最新 Schema？ |
| 跨设备同步 | 你在不同设备上工作时，如何获取最新版本？ |
| 跨贡献者协作 | 其他开发者 clone 仓库后，他们的 WorkBuddy 如何获取最新 Schema？ |
| AI 臃肿 | 在单个工作区里堆所有东西，会导致上下文过载 |

**解决方案**：用 Git 作为同步中枢，实现双向同步。

### 核心思想

```
┌─────────────────────────────────────────────┐
│         中央仓库（GitHub/Gitee）          │
│  - workbuddy-wiki-schema.md（版本化）    │
│  - 各知识库的 wiki/                      │
│  - 版本标签：v2026-04-26-04611bb        │
└──────────┬──────────────────┬─────────────┘
           │ Pull             │ Push
           ▼                  ▼
┌──────────────┐      ┌──────────────┐
│ 工作区A      │      │ 工作区B      │
│（你的设备1） │      │（你的设备2） │
└──────────────┘      └──────────────┘
           │
           ▼
┌──────────────┐
│ 贡献者A      │
│（其他开发者）│
└──────────────┘
```

**关键机制**：
1. Schema 独立版本化（不堆在单个工作区）
2. 所有工作区从同一个 Git 仓库拉取最新 Schema
3. 其他开发者的贡献也能合并进来
4. 每次迭代打标签，可回滚到任意版本

### 版本号规范

采用双版本号体系：

| 版本类型 | 格式 | 用途 | 示例 |
|----------|------|------|------|
| 人类可读版本 | `v<major>.<minor>.<patch>` | 快速理解迭代规模 | v0.4.0 |
| Git 精确版本 | `v<YYYY>-<MM>-<DD>-<short-hash>` | 精确回滚 | v2026-04-26-7eb7190 |

**升级规则**：
- `major`：破坏性变更（不兼容旧版本）
- `minor`：新增功能（兼容旧版本）→ 本次 v0.3.0 → v0.4.0
- `patch`：Bug 修复（兼容旧版本）

**自动打标签**：通过 Git post-commit hook 自动生成 Git 精确版本标签。

### 更新机制（手动触发，版本对比，用户确认）

> 汲取自 `github` skill：用 `gh` CLI 简化 git 操作，比原生 git 命令更高效。

**触发方式**：用户对 WorkBuddy 说"**检查 Schema 更新**"

**执行流程**：

```
1. gh repo sync owner/repo  （或 git fetch origin main）
   → 获取远程最新（gh 自动处理 upstream）

2. 读取本地版本号
   → 从 workbuddy-wiki-schema.md 开头读取

3. 读取远程版本号
   → gh api repos/owner/repo/contents/workbuddy-wiki-schema.md --jq '.head_commit.message'
   → 或直接读取远程文件头部

4. 版本对比
   ├─ 如果相同 → 告诉用户"已是最新版本"
   └─ 如果不同 → 继续下一步

5. 显示变更日志（用 gh，比 git log 更可读）
   → gh run list --repo owner/repo --limit 5
   → gh api repos/owner/repo/commits?path=workbuddy-wiki-schema.md --jq '.[].commit.message'

6. 显示详细差异
   → gh api repos/owner/repo/compare/<本地版本>...<远程版本> --jq '.files[]'
   → 或 git diff HEAD origin/main -- workbuddy-wiki-schema.md

7. 询问用户
   ├─ "是否应用此更新？"
   ├─ 是 → gh repo clone owner/repo --pull（或 git pull origin main）
   ├─ 否 → 记录跳过原因，下次再提醒
   └─ 查看详细差异 → 显示完整 diff

8. 更新后验证版本号，通知用户更新完成

9. 如需回滚
   → gh release view <tag> --repo owner/repo  （查看该版本详情）
   → git checkout <tag>  （本地回滚）
```

**`gh` CLI 优势（相比原生 git）**：
| 操作 | 原生 git | gh CLI |
|------|----------|--------|
| 查看远程文件内容 | `git fetch` + `git show` | `gh api repos/.../contents/file.md` |
| 查看 commit 历史 | `git log --oneline` | `gh api repos/.../commits?path=file` |
| 对比两个版本 | `git diff tag1 tag2` | `gh api repos/.../compare/tag1...tag2` |
| 查看 PR/Issue | 手动去网页 | `gh pr list` / `gh issue list` |

**安全设计**：
- 不自动执行（必须用户确认）
- 显示详细差异（`gh api` 或 `git diff`）
- 支持回滚（`git checkout <tag>`）

### 跨贡献者协作流程

其他贡献者 clone 仓库后：

```
1. 他们的工作区指向 clone 的目录
2. 启动时手动执行"检查 Schema 更新"
3. 他们在本地迭代，commit + push 回中央仓库
4. 我（元一）定期 review 并 merge 他们的贡献
5. 打新标签（v2026-04-27-xxx），通知所有工作区更新
```

### 与"知识库索引上传 + 全局统筹"的关系

本方案（Git-Based Schema Distribution）解决的是 **Schema 本身的版本化管理**。

在此基础之上，可以进一步构建 **跨知识库统筹架构**：

1. **各知识库从 Git/Gitee 获取新版本 Schema** → 自行迭代
2. **各知识库把自身索引上传到中央** → 索引格式：`{知识库名}-index.json`
3. **统筹层（元一）获取所有知识库的索引** → 构建全局视图
4. **统筹层根据全局视图，协调跨知识库链接** → 实现"流动趋效，均衡为度"

## 提示和技巧

- **Obsidian Web Clipper** 是一个将网络文章转换为 Markdown 的浏览器扩展。对于快速将资料来源放入你的原始集合非常有用。
- **本地下载图片。** 在 Obsidian 设置 → 文件和链接中，将"附件文件夹路径"设置为一个固定目录（例如 `素材/assets/`）。然后在设置 → 热键中，搜索"Download"以找到"下载当前文件的附件"并将其绑定到一个热键（例如 Ctrl+Shift+D）。在剪辑一篇文章后，按下热键，所有图片都会被下载到本地磁盘。这是可选的但很有用——它让 LLM 可以直接查看和引用图片，而不是依赖可能失效的 URL。注意，LLM 无法在一次传递中原生读取带有内联图片的 Markdown——变通方法是让 LLM 先读取文本，然后单独查看部分或所有引用的图片以获得额外的上下文。这有点笨拙但效果足够好。
- **Obsidian 的图谱视图**是查看 Wiki 形状的最佳方式——什么与什么相连，哪些页面是枢纽，哪些是孤儿。
- **Marp** 是一种基于 Markdown 的幻灯片格式。Obsidian 有一个插件支持它。

## 可视化（Canvas）

> 汲取自 `json-canvas` skill。**知识图谱的可视化表达，帮助发现信息孤岛和核心枢纽。**

### 理念

`.canvas` 文件是 Obsidian 的原生可视化格式（JSON 结构），可以在 Obsidian 中渲染为节点-边图形。Wiki 的「流动趋效」原则可以通过可视化来验证——哪些概念是核心枢纽（高连接度），哪些页面是信息孤岛（无反向链接）。

### 何时创建 Canvas 文件

| 场景 | 动作 |
|------|------|
| Wiki 概念页 ≥ 10 个 | 创建 `wiki/概念/概念图谱.canvas` |
| 发现信息孤岛或核心枢纽时 | 用 Canvas 标注节点颜色（红色=孤岛，绿色=枢纽） |
| 用户请求可视化某个主题的关系 | 创建专题 Canvas（如 `wiki/概念/Agent 架构图谱.canvas`） |

### Canvas 文件规范

- **存放位置**：与对应目录同级
  - `wiki/概念/概念图谱.canvas`
  - `wiki/实体/实体关系.canvas`
- **节点类型**：优先用 `file` 类型（链接到已有页面），而不是 `text` 类型（重复内容）
- **颜色编码**（对应四极飞轮）：
  - `"1"`（红）：信息孤岛，需要补充链接
  - `"4"`（绿）：核心枢纽，高频被引用
  - `"5"`（青）：新页面，刚创建尚未建立足够连接
- **更新时机**：每次 ingest 后，如果新增了概念页或实体页，更新对应 Canvas

### 与非 Obsidian 环境的兼容

- 用 `wiki/index.md` 的静态表格 + Obsidian 图谱视图替代

## 可选：CLI 工具

在某些时候，你可能想要构建小型工具来帮助 LLM 更高效地操作 Wiki。最明显的一个是 Wiki 页面上的搜索引擎——在小规模下索引文件就足够了，但随着 Wiki 的增长，你需要适当的搜索。[qmd](https://github.com/tobi/qmd) 是一个很好的选择：它是一个用于 Markdown 文件的本地搜索引擎，具有混合 BM25/向量搜索和 LLM 重排序，全部在设备上。它既有 CLI（所以 LLM 可以调用它），也有 MCP 服务器（所以 LLM 可以把它作为原生工具使用）。你也可以自己构建更简单的东西——LLM 可以帮助你根据需要对一个朴素的搜索脚本进行 vibe-coding。
