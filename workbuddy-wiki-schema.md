# WorkBuddy Wiki Schema

> **模式**：子知识库（只递交 `wiki/index.md` 和 `wiki/统筹/`）
> **版本**：v1.0.0（`v<YYYY>-<MM>-<DD>-<short-hash>`）
> **状态**：稳定（跨工作区/跨贡献者可部署）
> **思想**：元一思想 + 道德经十原则
> **变更**：道德经思想全部融入（10/10）；采用「文约而旨丰」风格重写

本文档是**地图**，非百科全书——按需扩展，不追求一次完美。

---

## 核心理念

WorkBuddy **增量构建并维护持久化 Wiki**——结构化、互链 Markdown 集合。知识编译一次即*保持最新*，非每次查询重新推导。

Wiki 是持久、具复合效应的产物：交叉引用已存，矛盾已标，每添一份资料，Wiki 即更丰富。

---

## 架构（三层）

1. **原始资料** — 不可变，只读
2. **Wiki** — WorkBuddy 生成的 Markdown 目录，完全拥有
3. **Schema** — 本文件，定义 Wiki 结构

> **关键洞察**：文件系统是天然协作面。Wiki 是 WorkBuddy 与你之间的协作面，所有知识持久化于文件系统。

---

## 顶层设计：元一思想

Wiki 是元一系统，四原则指导所有构建决策。

| 原则 | Wiki 构建含义 |
|------|----------------|
| **存续为体，形式为用** | Wiki 价值在於被使用，非为完美而存在 |
| **流动趋效，均衡为度** | 页面间链接通畅，信息查找成本低 |
| **意义生於博弈，固於认同** | Wiki 内容优先级由实际使用需求驱动 |
| **结构求稳，接口预变** | Schema 既稳定又可扩展 |

### 四极飞轮（健康度诊断）

```
存续为体(轴心) ↔ 流动趋效/均衡为度
           ↕
意义生於博弈/固於认同 ↔ 结构求稳/接口预变
```

### 接口决策规则

新需求 → 已有概念细化？→ 是 → 扩展现有页面 ✅  
　　　　↓ 否  
　　　　　全新概念/实体？→ 是 → 自下而上原则 ✅  
　　　　　　　　　　↓ 否  
　　　　　　　　　需改 Schema 核心？→ 先问：「能否通过接口扩展？」

## 操作

### 会话启动（4 步）

1. 读 `wiki/index.md` 了解规模
2. 读 `wiki/log/index.md` 了解近期操作
3. 读 `.workbuddy/memory/MEMORY.md` 获取跨会话事实
4. 读本文件确认约定

### Ingest（摄取）

**核心原则：一次只处理一篇资料（增量原则）。**

1. WorkBuddy 阅读原始资料
2. 与你讨论关键要点和侧重点
3. 创建 `wiki/来源/YYYY-MM-DD 标题.md` 摘要页
4. 更新相关实体页和概念页（自下而上原则）
5. 更新 `wiki/index.md`
6. 追加 `wiki/log/index.md`

> **结束时确保干净状态**：index、log、memory 皆已更新。

### Query（查询）

你针对 Wiki 提问。WorkBuddy 搜索相关页面，阅读，综合带引用的答案。**好答案可作为新页面归档回 Wiki**。

### Lint（健康检查 + 垃圾回收）

定期健康检查：
1. **悬空链接** — 扫描 `[[页面名]]`，确认目标存在
2. **双向链接** — 验证反向链接完整性
3. **Frontmatter** — 检查字段名拼写（`sources:`、`tags:`、`type:`）
4. **命名一致性** — 文件名与正文名称一致
5. **index.md 质量** — 简介有实质内容
6. **孤儿页面** — 无任何页面链接到它
7. **过时主张** — 对比原始资料，标记不准确描述

> **垃圾回收**：小额高频清理，非一次性大扫除。

## Wiki 写作规范

Wiki 读者是 WorkBuddy，非人类。写每页时问：**WorkBuddy 能直接从此页推理出完整含义吗？**

- 用 `[[页面名]]` 交叉引用；Frontmatter 用结构化字段
- 避免「如前所述」——智能体无法回溯上下文
- 图片用 `![[素材/图片名.png]]` 引用
- Obsidian 语法：`> [!info]`(Callouts)、`^block-id`(Block References)、`%% comment %%`(Comments)、`==高亮==`

## 道德经原则集成

### 知止不殆（停止迭代的信号）

> 「知足不辱，知止不殆，可以长久。」

**停止迭代信号**（满足任一即停）：
1. 当前版本稳定运行 ≥2 周，无新 bug
2. 新增功能使用频率 <10%
3. 复杂度已高到需 >5 分钟解释给新用户
4. 开始为「未来可能」而非「当下需要」添加功能

**版本升级决策**：当前版本稳定 + 无新需求 → 停止迭代 ✅

### 日损（定期做减法）

> 「为学者日益，闻道者日损。损之又损，以至于无为。」

**减法清单**（每季度一次）：
1. **功能减法** — 删使用频率 <10% 的功能
2. **工具减法** — 删不再使用的工具/Script
3. **文档减法** — 删过时或重复文档
4. **配置减法** — 删不再需要配置项

**核心**：新功能来了，先问「能不能通过减法解决？」能则做减法 ✅

### 无为而治（自动化流程设计）

> 「道常无为而无不为。」

让系统自动运行，减少人工干预。Wiki 维护成本应接近零。
- 自动化 Ingest、Lint、版本管理、备份
- 重复性工作优先自动化，高价值决策保持手动

### 辅万物之自然（辅助而非控制）

> 「是以圣人辅万物之自然，而不敢为。」

WorkBuddy 辅助用户，非控制工作流。
- Query 模式优先、用户确认机制、非侵入式建议
- **辅助 vs 控制判断**：辅助=提供信息/建议选项/执行确认后操作；控制=自动决策/强制执行/阻止用户选择

### 致虚极守静笃（保持简洁）

> 「致虚极，守静笃。」

保持 Schema 简洁，**文约而旨丰**（字少意多）。
- 新需求先问：「能不能用现有功能解决？」能则用现有，不能才新增。

### 大制不割（保持整体性）

> 「大制不割。」

保持 Wiki 整体性，避免碎片化。
- 相关概念放同一页面，避免过多小页面
- 用 Canvas 可视化整体结构，避免信息孤岛

### 反者道之动（逆向思维）

> 「反也者，道之动也。」

考虑逆向流程，验证系统鲁棒性。
- Ingest 前先想：如何撤销？如何回滚？
- 每次添加功能，先测试删除场景

### 柔弱胜刚强（轻量级工具链）

> 「柔弱胜刚强。」

优先轻量级工具链，提高可移植性。
- 优先用纯文本（Markdown），少用二进制格式
- 优先用标准工具（Git、Python），少用专有软件
- 工具链复杂度 ↑ → 可移植性 ↓ → 风险 ↑

### 小国寡民（去中心化知识库）

> 「小国寡民。」

避免单点故障，支持去中心化知识库。
- Wiki 可分布在多个目录/设备
- 用 Git 同步，非中心化服务器
- 单点故障 → 其他节点仍可工作

## Schema 版本化管理（Git-Based Distribution）

Schema 独立版本化，通过 Git 同步。

**版本号**：`v<major>.<minor>.<patch>`（人类可读）+ `v<YYYY>-<MM>-<DD>-<short-hash>`（Git 精确版本）

**升级规则**：`major`=破坏性变更，`minor`=新增功能，`patch`=Bug 修复

**更新机制**：用户说「检查 Schema 更新」→ 获取远程最新 → 版本对比 → 用户确认是否应用（安全设计：不自动执行）。

---

## 内容分发（Content Distribution）

Wiki 内容需分发到不同平台，并从平台反馈中汲取改进点。内容分发是 **接口扩展**，不修改 Schema 主体：

```
Wiki 内容 → 适配各平台格式 → 发布 → 收集反馈 → 汲取到 Wiki
```

---

## 可视化（Canvas）

`.canvas` 文件是 Obsidian 原生可视化格式。Wiki 概念页 ≥ 10 个 → 创建 `wiki/概念/概念图谱.canvas`。

---

## 为什么这有效

维护知识库的乏味部分是簿记（更新交叉引用、保持摘要最新）。人类放弃 Wiki 是因为维护负担增长快。WorkBuddy 不会感到无聊，不会忘记更新，可一次触及多个文件。Wiki 保持维护是因为维护成本接近零。
